# Локальний запуск і синхронізація між двома комп'ютерами

Ця інструкція для поточного репозиторію `web-site-home-structure-v1`, де:

- `ui/` - основний React + Vite UI-проєкт
- `v1.0` ... `v1.4` - wireframe-версії
- `v1.1-ui` - ранній окремий UI-прохід
- `docs/` - документація, логіка версій і handoff

Якщо тобі треба просто запустити React-проєкт локально, працюй у папці `ui/`.

## 1. Що встановити на новому комп'ютері

Перед першим запуском переконайся, що встановлено:

- `git`
- `node`
- `npm`

Рекомендовано перевірити:

```bash
git --version
node -v
npm -v
```

Якщо `node` або `npm` не встановлені, React UI не запуститься.

## 2. Перший запуск на новому комп'ютері

### Крок 1. Склонуй репозиторій

```bash
git clone https://github.com/Boychukd/web-site-home-structure-v1.git
```

### Крок 2. Перейди в проєкт

```bash
cd web-site-home-structure-v1
```

### Крок 3. Перейди в React UI

```bash
cd ui
```

### Крок 4. Встанови залежності

```bash
npm install
```

Це створить локальну папку `node_modules/`.

### Крок 5. Створи локальний env-файл

```bash
cp .env.example .env.local
```

Потім відкрий `ui/.env.local` і додай реальний ключ, якщо він тобі потрібен:

```bash
REACTBITS_LICENSE_KEY="your-license-key"
```

Важливо:

- `ui/.env.local` не пушиться в GitHub
- якщо на другому комп'ютері потрібен той самий ключ, його треба додати окремо вручну

### Крок 6. Запусти локальний сервер

```bash
npm run dev
```

Після цього Vite покаже локальну адресу, зазвичай щось типу:

```bash
http://localhost:5173/
```

Саме цю адресу відкривай у браузері.

## 3. Щоденний локальний запуск

Коли репозиторій вже є на комп'ютері, щоденний сценарій короткий:

```bash
cd web-site-home-structure-v1/ui
npm run dev
```

Якщо залежності вже встановлені, `npm install` повторювати не потрібно.

## 4. Як забрати останню версію з GitHub на другому комп'ютері

Це найважливіший сценарій для роботи з двома комп'ютерами.

### Варіант A. На цьому комп'ютері ще немає своїх незбережених змін

```bash
cd web-site-home-structure-v1
git pull origin main
cd ui
npm install
npm run dev
```

Пояснення:

- `git pull origin main` забирає останні зміни
- `npm install` варто запускати після оновлення, якщо змінився `package.json` або `package-lock.json`
- `npm run dev` піднімає UI локально

### Варіант B. На цьому комп'ютері вже є локальні незакомічені зміни

Спочатку перевір:

```bash
git status
```

Якщо є зміни, не роби `pull` навмання. Спершу обери один з безпечних шляхів:

#### Шлях 1. Закомітити і запушити свої зміни

```bash
git add .
git commit -m "wip: local progress"
git push origin main
git pull origin main
```

Це найкращий варіант, якщо зміни вже можна зберігати в репозиторії.

#### Шлях 2. Тимчасово сховати зміни

```bash
git stash
git pull origin main
git stash pop
```

Це підходить, якщо ти ще не готовий пушити код.

Якщо після `git stash pop` є конфлікти, їх треба розв'язати вручну.

## 5. Найшвидший спосіб передавати контекст між двома комп'ютерами

Щоб не згадувати "на чому я зупинився", використовуй один і той самий короткий ритуал перед завершенням сесії.

## 6. Рекомендований handoff-ритуал перед переходом на інший комп'ютер

Перед тим як закрити роботу на комп'ютері A:

### 1. Перевір статус

```bash
git status
```

### 2. Якщо зміни готові - закоміть і запуш

```bash
git add .
git commit -m "feat: short description"
git push origin main
```

### 3. Онови короткий контекст у `docs/handoff.md`

Записуй 3 речі:

- що вже зроблено
- що зараз у роботі
- який наступний крок

Можна прямо в такому форматі:

```md
## Current status

- Done: finished header cleanup in `ui/src/App.tsx`
- In progress: refining calculator block
- Next step: align CTA spacing and re-check mobile layout
```

### 4. Якщо код ще сирий, але контекст важливий

Навіть якщо не хочеш пушити незавершений UI, мінімум онови `docs/handoff.md`, щоб на другому комп'ютері одразу бачити:

- де саме ти працював
- які файли чіпав
- що треба зробити першим

## 7. Як відновити роботу на другому комп'ютері без втрати часу

На комп'ютері B:

```bash
cd web-site-home-structure-v1
git pull origin main
cat docs/handoff.md
cd ui
npm install
npm run dev
```

Порядок такий:

1. підтягнув останній код
2. відкрив короткий контекст
3. підняв React UI
4. продовжив з наступного кроку

## 8. Які файли читати, щоб швидко згадати контекст

Якщо після перерви треба швидко "увійти" назад у проєкт, дивись у такому порядку:

1. `docs/handoff.md` - поточний стан роботи
2. `docs/workflow.md` - що вважається wireframe, а що UI
3. `docs/version-map.md` - яка версія за що відповідає
4. `README.md` - коротка карта репозиторію

## 9. Що куди редагувати

Щоб не плутатися між wireframe і React UI:

- якщо працюєш над чистим React UI, редагуй `ui/`
- якщо тестуєш структуру, flow або чернеткову логіку, дивись `v1.x`
- не змішуй wireframe-правки з чистим UI без потреби

```

### На комп'ютері B перед стартом

```

Потім відкрий `docs/handoff.md` і продовжуй роботу.

## 12. Важливі примітки

- `node_modules/` не передається через GitHub, тому на кожному комп'ютері залежності стоять окремо
- `ui/.env.local` теж локальний, його треба налаштувати окремо на кожному комп'ютері
- якщо після `pull` змінився `package-lock.json`, краще ще раз виконати `npm install`
- якщо `npm run dev` не стартує після оновлення, перше що перевірити - чи ти точно знаходишся в папці `ui/`
